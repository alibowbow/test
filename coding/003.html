1. 버전관리가 왜 필요할까?

😱 버전관리가 없으면…

보고서_최종.html
보고서_최종_v2.html
보고서_최종_진짜최종.html
보고서_최종_진짜최종_이게진짜.html

언제 무엇을 왜 바꿨는지 모르고, 되돌리기도 어렵고, 여럿이 함께 고치면 누가 무엇을 덮어썼는지 알 수 없습니다.

✅ Git이 있으면…

  • ▸되감기 — 언제든 과거 어느 시점으로든 돌아갈 수 있어요(시간여행).
  • ▸기록 — 누가·언제·왜 바꿨는지 모든 변경이 남습니다.
  • ▸분기 — 망가뜨릴 걱정 없이 따로 떼어내 실험할 수 있어요.
  • ▸협업 — 여러 명이 충돌 없이 같은 프로젝트를 고칩니다.
💡 Git = 내 컴퓨터의 시간여행 저장 시스템  ·  GitHub = 그걸 인터넷에 올려 함께 쓰는 곳
🧩

2. 핵심 개념 4가지 (비유로)

카드에 마우스를 올리면(또는 탭하면) 뒤집혀 한 줄 정의가 나옵니다.

📦
저장소 (Repository)
프로젝트 폴더 = 게임 그 자체
프로젝트의 모든 파일 + 모든 변경 역사를 통째로 담는 상자. 줄여서 repo.
💾
커밋 (Commit)
게임의 세이브 포인트
이 순간의 스냅샷(저장점). "무엇을 했는지" 메모를 붙여 저장해요. 여기로 언제든 되돌아갈 수 있습니다.
🌿
브랜치 (Branch)
평행우주 / 작업 사본
원본(main)을 건드리지 않고 갈라낸 작업용 가지. 여기서 마음껏 실험하고, 잘 되면 합칩니다.
🔀
머지 (Merge)
평행우주를 다시 합치기
브랜치에서 한 작업을 다시 main에 합치는 일. 두 갈래의 변경을 하나로 모읍니다.
🔗 한 문장 정리: 저장소 안에서 커밋으로 저장점을 쌓고, 위험한 작업은 브랜치로 떼어내 한 뒤, 다 되면 머지로 합칩니다. 그 "합쳐도 될까요?"라는 요청이 바로 다음에 나올 PR입니다.
🗺️

3. 전체 흐름 따라가기 (클릭!)

아래 다음 → 버튼을 누를 때마다 한 단계씩 진행됩니다. 파란 선은 main, 보라 선은 작업 브랜치예요.

단계 1 / 8

시작 — 안정된 main

main 브랜치에는 잘 작동하는 코드가 있습니다. 모두가 믿고 쓰는 진짜 버전이에요.

🖱️ GitHub 화면에서 PR 만드는 법 (3단계)

브랜치를 push하면 GitHub가 보통 노란 배너로 "Compare & pull request" 버튼을 띄워줍니다.

STEP 1
Pull requests 탭 → New
base: main ← compare: 내 브랜치 선택
STEP 2
제목·설명 작성 → Create
무엇을 왜 바꿨는지 적어요. 그러면 리뷰·CI가 돌아갑니다.
STEP 3
Merge 버튼 클릭
초록 Merge pull request → 끝! 브랜치는 Delete로 정리.
⚖️

4. "무조건 main에 merge" 해도 될까?

결론부터: 혼자 쓰는 작은 저장소라면 괜찮습니다. 다만 무엇을 잃고 있는지는 알아두면 좋아요. 아래 두 방식을 비교해 보세요.

🅐 main에 바로 작업/머지

  • + 빠르고 단순함 — 혼자 메모/문서 저장소엔 충분
  • − 작업하다 깨지면 main이 곧바로 깨짐 (안전망 없음)
  • − 합치기 전 리뷰·자동검사(CI)를 끼울 자리가 없음
  • − 여러 명이면 서로 덮어쓰기·충돌 위험 ↑

🅑 브랜치 → PR → 머지

  • + main은 항상 작동하는 상태로 보호됨
  • + 합치기 전에 검토·테스트·토론 가능
  • + 문제가 생겨도 그 PR만 되돌리면 끝(롤백 쉬움)
  • − 단계가 조금 늘어남 (혼자면 과할 수 있음)

👉 그래서, 이렇게 쓰면 됩니다

📝
혼자 메모/문서 저장소
main 직접도 OK. 대신 커밋 메시지만 잘 남기세요.
🧪
큰 실험·위험한 변경
혼자라도 브랜치로 떼어내면 안전. 망치면 버리면 됨.
👥
남과 함께 / 공개 프로젝트
PR 필수. 리뷰·CI·기록이 협업의 핵심.

팁: GitHub 저장소 설정에서 Branch protection을 켜면 "main에 직접 push 금지 / PR 필수"를 강제할 수 있어요. 습관을 바꾸고 싶을 때 유용합니다.

⌨️

5. 명령어 치트시트 (복사해서 쓰세요)

브랜치 → PR → 머지 흐름을 그대로 따라가는 순서입니다. 기능추가는 원하는 브랜치 이름으로 바꾸세요.

git checkout -b 기능추가

① main에서 새 브랜치를 만들고 그쪽으로 이동

git add . git commit -m "무엇을 했는지 메모"

② 변경을 담아 저장점(커밋) 만들기

git push -u origin 기능추가

③ 브랜치를 GitHub에 올리기 (그 뒤 화면에서 PR 생성)

git checkout main git pull origin main

④ 머지 후, 내 main을 최신 상태로 당겨오기

git branch -d 기능추가

⑤ 끝난 작업 브랜치 삭제 (로컬)

git status git log --oneline --graph --all

🔎 현재 상태 보기 / 커밋 그래프를 터미널에서 그려보기

📖

6. 용어 사전

Repository (레포)

프로젝트 + 모든 변경 역사를 담는 저장소.

Commit (커밋)

한 시점의 스냅샷(저장점). 메모를 붙여 저장.

Branch (브랜치)

원본을 건드리지 않는 갈래(작업 사본).

Merge (머지)

갈라진 작업을 다시 하나로 합치기.

Pull Request (PR)

"내 브랜치를 합쳐도 될까요?"라는 제안+리뷰 공간.

Clone (클론)

원격 저장소를 내 컴퓨터로 통째로 복제.

Push / Pull

내 변경을 올리기(push) / 남의 변경 받기(pull).

Fork (포크)

남의 저장소를 통째로 내 계정에 복사해 시작.

Conflict (충돌)

같은 줄을 둘이 다르게 고쳐 합칠 때 생기는 충돌. 손으로 골라줘야 함.

CI

올린 코드를 자동으로 빌드·테스트해주는 검사 로봇.

Rebase (리베이스)

커밋 역사를 깔끔하게 재정렬하는 고급 머지 방식.

Main / Origin

main = 기본(대표) 브랜치 · origin = 기본 원격(GitHub) 별칭.

🌳 핵심 한 줄: main은 건드리지 말고, 브랜치에서 일하고, PR로 검토받아, 머지로 합친다.
혼자라면 단순하게, 함께라면 PR로 — 상황에 맞게 고르면 됩니다.
